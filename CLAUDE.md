## Project Overview

This is an Express.js REST API template built with TypeScript, featuring comprehensive authentication (local + OAuth), database management via Drizzle ORM, and production-ready middleware configuration.

## Development Commands

### Running the Application

- `npm run dev` - Start development server with hot reload (uses tsx + nodemon)
- `npm start` - Run production build (requires prior build)
- `npm run build` - Compile TypeScript to JavaScript in `dist/` directory
- `npm run watch` - Watch mode for TypeScript compilation

### Database Operations

- `npm run db:generate` - Generate Drizzle migrations from schema changes
- `npm run db:migrate` - Apply pending migrations to database
- `npm run db:push` - Push schema directly to database (bypass migrations)
- `npm run db:pull` - Pull schema from database
- `npm run db:studio` - Open Drizzle Studio for database inspection

### Code Quality

- `npm run format` - Format code with Prettier
- `npm run format:check` - Check code formatting without changes

## Architecture

### File Naming Conventions

This codebase follows **strict kebab-case naming** for all files with descriptive suffixes indicating file purpose:
- Controllers: `*.controller.ts` (e.g., `users.controller.ts`)
- Services: `*.service.ts` (e.g., `auth.service.ts`)
- Repositories: `*.repository.ts` (e.g., `users.repository.ts`)
- Routes: `*.route.ts` (e.g., `auth.route.ts`)
- Middlewares: `*.middleware.ts` (e.g., `validate.middleware.ts`)
- Validators: `*.validator.ts` (e.g., `users.validator.ts`)
- Configurations: `*.config.ts` (e.g., `app.config.ts`)

### Entry Point & Server Setup

- `src/index.ts` - Application entry point, starts Express server on configured PORT with graceful shutdown (SIGINT/SIGTERM)
- `src/app.ts` - Express app configuration with complete middleware stack and route mounting
- `src/env.ts` - Environment variable validation using Zod schema with runtime type safety

### Configuration Files

- `src/config/app.config.ts` - Centralizes CORS, Helmet, and Compression middleware configuration
- `src/config/passport.config.ts` - Passport.js strategies for Google and Facebook OAuth
- `drizzle.config.ts` - Drizzle ORM configuration for database migrations

### Third-Party Client Integrations

- `src/clients/cloudinary.ts` - Cloudinary SDK configuration for cloud storage/media management
- `src/clients/google-oauth.ts` - Google OAuth2Client initialization

### Application Constants

- `src/constants/api-messages.ts` - Centralized API response messages (validation, error, success messages)
- `src/constants/auth.ts` - Authentication constants (user roles, provider types, verification types, cache key prefixes, OAuth callbacks)
- `src/constants/expiry.ts` - Token and verification expiry durations (access token, refresh token, email verification, password reset)

### Type Definitions

- `src/types/index.ts` - TypeScript type definitions and Express module augmentation (adds `user` property to Request interface)

### Path Aliases

TypeScript path alias `@/*` maps to `src/*`. After build, use `tsc-alias` to resolve aliases in compiled output.

### Request Flow

1. Request enters through Express middleware stack (helmet, compression, CORS, morgan, body parsing)
2. Routes mounted at `/api/v1` prefix (see `src/routes/`)
3. Validation middleware (`validate.middleware.ts`) uses Zod schemas from `src/validators/`
4. Authentication middleware (`auth.middleware.ts`) verifies JWT tokens for protected routes
5. Controllers handle business logic and call services
6. Services coordinate between repositories
7. Repositories abstract database operations using Drizzle ORM
8. Responses standardized via `ApiResponse` class
9. Errors caught by `errorMiddleware` and formatted consistently

### Database Layer

- **Database Connection**: `src/database/db.ts` - PostgreSQL via `postgres` driver + Drizzle ORM
- **Schema Location**: `src/database/schema/` - Drizzle schema files:
  - `users.ts` - User accounts table schema
  - `sessions.ts` - Refresh token sessions table schema
  - `verifications.ts` - Email verification and password reset codes table schema
  - `user-locations.ts` - User login location tracking table schema
  - `relations.ts` - Drizzle ORM relational mappings (one-to-many, many-to-one)
  - `index.ts` - Exports all schemas for centralized import
- **Migrations**: Stored in `src/database/migrations/` (auto-generated by drizzle-kit)
- **Repositories**: Each entity has a dedicated repository in `src/repositories/`:
  - `users.repository.ts` - User CRUD operations and queries
  - `sessions.repository.ts` - Session management operations
  - `verifications.repository.ts` - Verification code operations
  - `user-locations.repository.ts` - Location tracking operations

### Authentication System

- **JWT-based**: Access tokens (JWT, 10h expiry) + Refresh tokens (UUID stored in sessions table, 30d expiry)
- **Local Auth**: Email/password with bcrypt hashing (10 rounds)
- **OAuth Providers**: Google and Facebook via Passport.js + Google OAuth2Client
- **Email Verification**: OTP codes (15m) and verification links (24h) stored in verifications table
- **Password Reset**: Token-based (15m) with OTP fallback (10m) and reset links (1h)
- **User Roles**: Admin, User, Moderator (defined in `constants/auth.ts`)
- **Middleware**:
  - `src/middlewares/auth.middleware.ts` - JWT token verification and user authentication
  - `src/middlewares/multer.middleware.ts` - File upload handling

### Utility Modules

- `src/utils/api-error.ts` - ApiError class for standardized error handling with HTTP status codes
- `src/utils/api-response.ts` - ApiResponse class for consistent JSON responses
- `src/utils/async-handler.ts` - Wrapper for async route handlers to catch errors automatically
- `src/utils/logger.ts` - Winston logger configuration for application logging
- `src/utils/http-status.ts` - HTTP status code constants
- `src/utils/get-device-info.ts` - Extract device details (browser, OS, platform) from user-agent
- `src/utils/get-location.ts` - IP geolocation using fast-geoip library
- `src/utils/password.ts` - Password hashing and comparison using bcrypt
- `src/utils/send-email.ts` - Nodemailer email sending functionality
- `src/utils/handle-postgres-error.ts` - PostgreSQL error mapping to user-friendly messages
- `src/utils/helpers.ts` - General utility functions (`parseTimeToMs`, `getUpdatedFields`, `isEmptyObject`)
- `src/utils/oauth-helpers.ts` - OAuth flow helper functions
- `src/utils/repository-helpers.ts` - Database repository helper utilities
- `src/utils/schema-helpers.ts` - Zod schema creation helpers (`createSchema` function)
- `src/utils/validation-helpers.ts` - Request validation utility functions

### Environment Variables

All environment variables validated in `src/env.ts` using Zod. Required variables include:

- Database: `DATABASE_URL`
- JWT secrets: `ACCESS_TOKEN_SECRET`, `REFRESH_TOKEN_SECRET`
- OAuth credentials: Google and Facebook client IDs/secrets
- SMTP configuration for email sending
- Cloudinary for file uploads (if used)
- Redis configuration (if caching implemented)

See `.env.example` for complete list.

### Error Handling

- All errors flow through `src/middlewares/error.middleware.ts`
- PostgreSQL errors mapped to user-friendly messages via `handlePostgresError()`
- Custom ApiError class supports common HTTP status codes
- Development mode includes stack traces in error responses

### Middleware Stack Order (src/app.ts)

1. Helmet (security headers)
2. JSON/URL-encoded body parsers
3. Compression
4. CORS (with origin validation)
5. Morgan (HTTP request logging)
6. Routes at `/api/v1`
7. 404 handler for undefined routes
8. Global error middleware (must be last)

## Key Patterns

### Repository Pattern

Repositories encapsulate all database queries. When adding new entities:

1. Create schema in `src/database/schema/`
2. Export from `src/database/schema/index.ts`
3. Create repository in `src/repositories/`
4. Use repository methods in services

### Validation Pattern

Request validation uses Zod schemas defined in `src/validators/`. Apply via `validate()` middleware:

```typescript
router.post('/endpoint', validate(mySchema), controller.handler);
```

### Response Patterns

- Success: `ApiResponse.success(res, data, message)`
- Created: `ApiResponse.created(res, data, message)`
- Error: `throw ApiError.badRequest(message)` (caught by error middleware)

### Async Handlers

Use `asyncHandler` wrapper from `src/utils/async-handler.ts` to automatically catch async errors in controllers.

## Adding New Routes

1. Create route file in `src/routes/`
2. Import and mount in `src/routes/index.ts`
3. Create validator schemas in `src/validators/`
4. Implement controller in `src/controllers/`
5. Add service layer in `src/services/` if complex logic needed
6. Create repository if new database entity involved

## Session Management

- Sessions table stores refresh tokens with expiry timestamps
- Each login creates new session with device/location info
- Logout invalidates specific session
- Logout-all invalidates all user sessions
- Expired sessions should be cleaned up periodically (implement cron job if needed)
